# 分阶段验证开发

## 核心规则

不要把“代码已经写完”当成任务完成。只有在验证报告已经生成、用户已经审阅、用户明确确认通过，并且进度追踪文件已经记录后，模块或阶段才算真正完成。

一步一步来，不怕麻烦，就怕做错。不跳步、不跨阶段、不擅自替用户做关键决策。

## 不可违反的规则

- 开发前必须先编写或更新设计文档。
- 设计文档是项目的单一真相源。
- 技术选型、架构、外部服务、部署方式等关键决策必须逐项向用户确认。
- 每个模块都必须生成 Markdown 验证报告，辅助模块也不能跳过。
- 每份验证报告生成后必须停止，等待用户确认。
- 用户确认通过后，才能把进度状态更新为“已确认通过”，并继续下一个模块。
- 所有外部依赖必须通过 provider 抽象隔离。
- 外部服务失败时必须有降级逻辑，并且必须验证降级逻辑。
- 遇到阻塞必须及时通知用户，不允许自行猜测并继续。

禁止：

- 把测试通过等同于用户验收通过。
- 在用户未确认验证报告前继续开发下一个模块。
- 未经确认就替换技术方案、架构、provider、模型、数据库、部署目标或外部服务。
- 隐藏警告、失败样本、数据缺失、异常结果或降级状态。
- 把多个阶段合并成一次大开发。

## 项目启动流程

1. 建立协作文档体系：
   - `docs/design.md`：设计文档，单一真相源
   - `docs/progress.md`：进度追踪文件
   - `docs/api-contract.md`：API 契约文档
   - `docs/environment.md`：环境搭建指南
   - `docs/decision-log.md`：关键决策日志
   - `docs/risks-and-optimizations.md`：风险与优化项
   - `reports/verification/`：验证报告目录
2. 先写设计文档，再写代码。
3. 逐项向用户确认关键决策：
   - 技术栈
   - 架构和模块边界
   - 外部服务和 provider 模式
   - 数据来源和存储方式
   - 部署方式
   - 阶段计划和验收标准
4. 每个决策确认后，更新设计文档和决策日志。
5. 初始设计和阶段计划确认后，才开始实现。

创建文档时读取 `references/document-templates.md`。

## 恢复上下文流程

继续一个已有项目时，先读取：

- 设计文档
- 进度追踪文件
- API 契约文档
- 决策日志
- 最新验证报告
- 当前风险和阻塞项

然后先向用户总结当前状态：

- 当前阶段
- 当前模块
- 最近一次已确认通过的验证
- 待确认决策
- 阻塞或风险
- 下一步建议动作

## 模块开发流程

每个模块都按这个流程执行：

1. 确认该模块是进度文件中允许开始的下一个任务。
2. 只实现当前模块和它的直接前置依赖。
3. 运行模块，生成必要的中间产物。
4. 运行模块级测试或验证命令。
5. 在 `reports/verification/` 下生成 Markdown 验证报告。
6. 把报告状态标记为 `PASS`、`WARN`、`FAIL` 或 `BLOCKED`。
7. 把报告交给用户审阅，然后停止。
8. 只有用户明确确认通过后，才能继续。
9. 将 `docs/progress.md` 中对应任务从“已完成”更新为“已确认通过”。

如果报告是 `FAIL`，必须修改后重新生成报告。
如果报告是 `WARN`，必须说明风险，并让用户选择：接受、立即修复、还是记录为优化项。
如果报告是 `BLOCKED`，必须向用户说明需要什么决策、凭证、数据、服务或环境变更。

生成报告时读取 `references/verification-report-template.md`。

## 阶段管理

每个阶段必须明确：

- 阶段目标
- 包含模块
- 依赖顺序
- 完成标准
- 验收指标
- 必要验证报告
- 已知风险

阶段内按依赖顺序开发：

1. 基础模块
2. 依赖模块
3. 集成模块
4. 阶段级验证

当前阶段未被用户确认通过前，不得开始后续阶段。

## Provider 架构

所有外部依赖都必须封装在 provider 接口后面。优先预留三种模式：

- `local`：本地实现，用于第一阶段快速跑通
- `api`：第三方或云端 API 实现
- `self_hosted`：内部部署或私有化实现

业务逻辑只能依赖 provider 接口，不能直接依赖具体外部服务。

每个 provider 都要记录：

- 接口契约
- 当前启用模式
- 配置方式
- 降级逻辑
- 超时与重试策略
- 错误暴露方式
- 切换步骤
- 回滚步骤

## 验证策略

使用能证明当前模块正确性的最轻量可靠验证。

必须覆盖的验证类型：

- 数据验证：将读取结果和原始输入对比，报告丢失、错配、重复和解析错误。
- 功能验证：用标准测试集独立验证当前模块。
- 准确率验证：涉及抽取、分类、匹配、检索、生成时，使用自命中或标准答案验证。
- 降级验证：模拟 provider 不可用，确认降级逻辑生效。
- 集成验证：阶段完成前验证模块之间的契约。

每份验证报告都必须包含可复现命令、输入路径、输出路径、样本、指标、警告和异常样本。

## 进度状态

任务状态只能使用：

- `待开始`
- `进行中`
- `已完成`
- `已确认通过`
- `阻塞`

只有用户可以把“已完成”的模块确认为“已确认通过”。

## 决策处理

每个重要决策都要记录：

- 决策事项
- 可选方案
- 推荐方案
- 用户选择
- 时间
- 理由
- 影响范围
- 回滚或替换路径

涉及不可逆、高成本、安全敏感或影响架构的选择时，必须先问用户。

## 问题处理

- 正确性、数据丢失、安全、契约失败问题必须在继续前修复。
- 性能和资源问题可以记录为优化项，除非它阻塞验证。
- 外部服务失败必须触发降级逻辑，并写入验证报告。
- 缺少凭证、需求不清、数据不可访问、服务不可用都属于阻塞。

## 对用户汇报时必须说明

- 实现了什么
- 用什么命令或流程验证了
- 验证报告在哪里
- 当前报告状态是什么
- 继续前需要用户确认什么

不要把确认请求藏在长篇说明里。每个模块完成后，都要明确请求用户审阅验证报告并确认是否通过。
